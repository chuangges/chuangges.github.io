微服务技术栈

发表于 2022-11-10 更新于 2023-04-19 分类于 前端技术栈
本文字数： 19k 阅读时长 ≈ 17 分钟
微服务、微前端、中间件、NodeJS 微服务

一、微服务

系统架构

分布式通过服务拆分解耦的方式实现了性能优化和按需更新但不能解决并发性和高可用性问题，集群则解决了系统的并发问题还可以在服务器宕机时正常运转，但是集群通过一台主机接收请求然后分发请求到部署相同服务的不同服务器实现负载均衡，本质上还是一个单体应用架构。具体设计时一般会结合分布式和集群的特点，即对拆分的子模块进行集群部署。

  • 单体应用：所有模块部署在一个服务器。优点是开发成本低、小型项目的首选，缺点是复杂项目不易开发和维护、性能扩展只能通过扩展集群结点但是有瓶颈和较高成本。集群可以提升性能但也有用户访问哪个服务器和数据共享的问题，Niginx 可用来反向代理用户请求地址并负载均衡完成用户请求的转发，Redis + Tomcat 则是实现数据共享的集群方案，但是这样的高并发问题导致数据库的压力过大，数据库优化方案主要有主从复制、Redis 缓存等。
  • 垂直应用：前后端分离的 MVC 分层架构，比如水平拆分根据层次拆分为工具库、数据持久、业务逻辑、后台管理、web 访问等模块并根据访问量合理部署，垂直拆分则根据功能拆分为商品、支付、购物车等独立模块。优点是前后端可以采用不同的技术开发、减轻了服务器的压力，缺点是某个模块更新时需要重新部署整个应用、不同服务器部署的模块之间无法交互。
  • 分布式架构：根据业务拆分为多个独立模块并通过 RPC、Http、HttpClient 等底层技术解决模块之间的调用问题。但是当服务越来越多，容量评估、小服务浪费资源等问题逐渐显现，此时需增加一个调度中心 SOA 基于访问压力实时管理集群容量并提高集群利用率。
  • SOA 面向服务的架构：一种以粗粒度(表示类别)、松耦合的服务为中心的架构。实现方式主要有中心化的 ESB、去中心化的微服务。ESB 总线通过协议转换、消息路由、事务控制等底层技术对不同技术栈开发的异构系统实现连接通信和统一管理，将所有服务集成为一个松耦合的分布式系统，但是这种单体架构模式也带来了环境搭建困难和性能瓶颈等问题。
  • 微服务架构：基于 SOA 理念针对业务实现细粒度(表示对象类别的实例)的组件化和服务化，通过RPC/REST协议对松耦合的微服务实现管理和整合，ESB 则是为各自为政的微服务们创造了一个中心化的政府。这种架构支持跨语言开发和独立部署，解决了复杂应用的快速开发问题，但也存在跨服务更改、调用链追踪、配置管理等问题，适用于长期迭代更新的项目。

开发技术

单体应用拆分出的独立服务就是微服务，它们相互调用会形成一个中心的服务集群，不同中心则是分布式，为了统一管理和配置而有了注册中心与配置中心、用户访问服务时在服务集群中处理负载均衡和请求路由的 API 网关，服务之间的通信机制则基于 对内的 REST、对外的 RPC、异步的消息队列，为了数据库高并发查询数据时更安全快速而产生了分布式缓存和分布式搜索，为了在调用链某一环节出错时不影响整体而提出的方案有重试机制、限流、熔断机制、负载均衡、降级，为了排查服务故障而引入分布式日志服务来记录请求信息并通过系统监控、链路追踪来监控整个系统运行并在异常时快速反应，最后通过 Jenkins + docker + kubernetes/rancher 实现自动化部署。一个微服务项目的架构和主要技术栈如下：

  • 架构方案：聚合器(将多个服务合并处理为一个新服务)、代理(服务转发和数据转换)、链式(同步调用的微服务链条)、分步式(支持同时调用多个微服务链)、数据共享(支持部分微服务共享数据库和缓存而存在强耦合)、异步消息(通过消息队列中间件实现异构系统的连接通信)。
  • 远程调用：同步消息通过基于 HTTP 对接第三方服务的接口规范 REST(对外暴露接口)、基于 TCP/UDP 内部高效通信的远程调用协议 RPC(对内直接调用)，异步通信则通过消息队列。
  • RPC 框架：Java 语言的 Dubbo、Motan、SpringBoot、SpringCloud，NodeJS 语言的 Seneca、Hapi、Restify、LoopBack，支持跨语言平台开发的 gRPC、Thrift。
  • 主要技术栈及其常用中间件：配置管理 SpringCloudConfig Nacos、注册发现 Nacos Eureka Zookeeper Consul、API网关 SpringCloudGateway Zuul、熔断限流 Hystrix Envoy Sentinel、服务监控 Zabbix Nagios Metrics Specatator、负载均衡 Ribbon Nginx、消息队列 Kafka RabbitMQ ActiveMQ、链路追踪 Zipkin Brave Dapper、自动化部署 Jenkins Docker Rancher Kubernetes。

二、微前端

基于后端微服务理念由多个独立交付的多个前端应用集成为一个整体应用拆分为的架构风格，核心原则为独立运行、独立部署、独立开发。微服务架构将后端服务拆分为用户、订单、支付、消息等可以相互调用的微服务，微前端应用将前端拆分为不同技术栈的 React 微应用、Vue 微应用、共享模块等。

系统架构

实现方案

  • Nginx 代理转发：服务端通过反向代理转发微应用的路由，切换时会触发浏览器刷新。
  • iframe 浏览器原生隔离方案：主应用通过 iframe 加载的子应用具备天然的沙盒机制，缺点是不支持 SEO、不共享变量、每次进入微应用都会重建一次浏览器上下文而无法前进后退。
  • WebComponent 组件化：通过自定义元素来加载不同微应用，通过附加到元素来实现功能并不暴露到全局的 ShadowDOM 实现隔离，存在改造成本大、组件通信、浏览器兼容等问题。
  • Module Federation 模块联邦：用于将多个独立构建组成一个应用程序的 Webpack5 新增功能。优点是开箱即用、组件共享、去中心化，缺点是无法沙箱隔离(需借助工具)、代码封闭性高(需通过 npm 管理)、拆分粒度需权衡(无法消除无用代码)、依赖前置而加长加载时间。
  • 中心基座方案：基座应用负责全局资源的加载运行、微应用的集成管理、路由映射、消息管理等，微应用独立开发并注册到基座。优点是支持 SEO、无感切换，缺点是沙箱不隔离。
  • 常用框架：新框架的后台管理系统可选择 Qiankun/MicroApp/Garfish，老旧框架的项目可选择 Qiankun/MicroApp，SEO 项目可选择 Piral/PuzzleJs。
  • Single-spa：将多个单页面应用聚合为一个整体应用的最早框架，可通过多种技术栈开发应用并使用公共路由实现切换，缺点有无通信机制、不支持 JS 沙箱、样式冲突、无法预加载。
  • Qiankun：阿里系开源的的 single-spa 优化版，基于中心基座方案但弥补了上述缺点。
  • MicroApp：京东一款基于 WebComponent 实现的开源框架，接入成本最低并且功能完善。
  • EMP：欢聚时代基于 Module Federation，支持跨应用状态共享、远程调用共享模块等。
  • Garfish：字节跳动开源，主要解决跨团队协作、技术体系多样化、应用日益复杂化等问题。
  • Wujie：腾讯基于 WebComponent + iframe，极大降低了主应用使用和微应用适配的成本。
  • Icestark：阿里面向大型系统，主应用管理微应用的注册和渲染而实现了系统的彻底解耦。

主要问题

  • 应用隔离：JS 沙箱隔离 (Sandbox 沙箱指一个独立运行的虚拟环境，具有变量隔离、独立路由、多个实例等特点)、CSS 样式隔离 (BEM 命名约定、CSS Module、CSS-in-JS)。
  • 跨应用通信：自定义事件 (低耦合)、React/Vue 全局状态、发布订阅模式、地址栏传参。
  • JS 沙箱机制：标签 iframe、属性 diff、ES6 Proxy、提案API ShadowRealm。

三、中间件

一种用于连接操作系统、数据库等系统软件与应用软件，并管理计算资源和网络通信的中间软件。它为应用提供了开箱即用的业务服务和标准的通信接口，屏蔽了底层实现逻辑而让开发者专注于业务，解决了异构网络环境下分布式应用的通信问题，常作为服务器或插件被部署到应用开发工具。应用可以直接和用户交互但必须通过中间件和底层系统实现交互。

主要分类

  • 基础中间件：用于应用通信、请求处理、web 应用构建等。主要有交易中间件 (用于银行业务等需要处理大量并发进程的联机交易系统)、消息中间件 (用于应用之间的异步网络通信)、应用服务器中间件 (用于浏览器和数据库之间业务处理的开发工具和运行平台)。
  • 集成中间件：用于数据库和应用等异构系统之间进行资源整合，构建一个分布式企业应用。
  • 行业领域应用平台：基于以上中间件针对特定行业开发文件管理、数据共享、云计算等功能。

常用实例

  • 路由与web服务器：处理和转发其他服务器数据的服务器。Nginx、Tengine、VipServer。
  • RPC 框架：微服务时代的远程服务调用框架。grpc、Thrift、HSF、Dubbo、SOFA-RPC。
  • 消息中间件：分布式应用之间发送和接收消息的软件。Kafka、RabbitMQ、RocketMQ。
  • 缓存服务：分布式的高速数据存储层，一般是内存存储。Tair、Redis、Memcached。
  • 配置中心：统一管理各个项目中所有配置。阿里 Nacos、携程 Apollo、百度 Disconf。
  • 分布式事务：事务的参与者、支持事务的服务器、资源服务器以及事务管理器，分别位于不同的分布式系统的不同节点。阿里 seata、腾讯 DTF。
  • 任务调度：分布式环境下提供定时、任务编排、分布式跑批等功能的系统。阿里 SchedulerX、业界 xxl-job、当当 elastic-job、有赞 TSP。
  • 数据库层：支持弹性扩容和分库分表的 TDDL、数据库连接池 Driud、Binlog 同步的 Canal。

消息中间件

缺点：系统可用性降低、系统稳定性降低、分布式一致性问题。

  • 复杂链路的异步调用：如果大量并发请求不需要服务器及时响应时则可将消息缓存后按顺序处理，比如用户注册(发布)、数据入库、消息队列、异步发送注册短信。
  • 复杂系统的解耦：将服务拆分，比如用户下单(发布)、消息队列、库存系统(订阅)处理业务。
  • 瞬时高峰的削峰处理：商品秒杀活动时过滤掉大多数无效请求后转发，比如用户下单(发送)、消息队列(根据实际情况限制人数等)、订阅消息根据秒杀规则处理业务。
  • 分布式应用的通信：根据应用使用的开发语言选择合适接口实现通信并对数据进行处理和转发，比如服务 A (发送消息)、消息队列、服务 B (订阅消息)。

JS 实现

每一个中间件执行结果传递给下一个中间件并等待其结束以后又回到当前执行逻辑，缺点主要有系统可用性降低、系统稳定性降低、分布式一致性问题。

  • axios：最简单，直接 promise 链式调用。
  • redux：通过 reduce 静态生成，将中间件的执行结果作为上一个中间件的 next 参数。
  • express、Redux：next 通过 redux 静态生成并已确定执行顺序，express 则是动态执行。
  • express、koa：通过递归动态执行，区别在于 koa 通过 return promise 形成洋葱模型，expres 则在调用时没有 return，最终执行的是 await undefind 而不是 await promise。
  • compose 为什么必须返回 promise.resolve：防止只有请求中间件且为同步函数时报错。
  • redux 中间件必须返回函数：redux 多了一个组合的过程，组合时需要将中间件的执行结果而不是函数来作为上一个中间件的 next 参数，否则传给中间件就会报错。

四、NodeJS 微服务

构建的主要优势有：单线程+异步 (通过事件循环来执行代码而实现非阻塞机制)、事件驱动 (适用于实时应用的发布订阅或观察者模式)、快速和高度的可扩展性 (运行环境基于引擎 V8 JavaScript Engine 而支持快速执行和处理大量并发请求)、易于开发 NodeJS 框架抽象了大部底层系统而可以轻松创建微服务而不会导致大量重复代码。
